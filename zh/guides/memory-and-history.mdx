QitOS 刻意把两类能力拆开：

- **History**：每步发给 LLM 的对话上下文。它是短期的、消息级别的，并由 `HistoryPolicy` 控制。
- **Memory**：跨步积累的语义记录存储，可以按 window、summary、vector 或 file 等策略检索。

两者都不是必选项。如果你两者都不传，Engine 会使用内部默认的 24-message window history，并且不启用 memory adapter。

## 正确的心智模型

研究者常常把“context”混成一个概念，但在 QitOS 里它被故意拆开：

| 关心的问题 | QitOS 原语 | 常见用途 |
|---|---|---|
| 下一步模型会看到什么 | `History` + `HistoryPolicy` | prompt 连续性 |
| agent 想跨步记住什么 | `Memory` | findings、artifacts、prior observations |
| 当上下文太大时怎么办 | `TokenBudgetSummaryHistory` / `CompactHistory` | 长时运行控制 |

这正是 QitOS 处理 long-running agents 最实用的设计之一：你可以单独调消息窗口，而不必重写 memory；也可以替换 memory 策略，而不影响 prompt 协议。

## History

`History` 存储的是 `HistoryMessage`（`role`、`content`、`step_id`）。每次 LLM 调用前，Engine 会先追加 user message，调用后再追加 assistant message。真正发送给模型时，再由 `HistoryPolicy` 决定要取哪一段窗口。

### HistoryPolicy

`HistoryPolicy` 位于 `qitos.core.history`，负责决定本次模型调用要看到哪些消息：

```python
from qitos.core.history import HistoryPolicy

policy = HistoryPolicy(
    roles=["user", "assistant"],
    max_messages=24,
    step_window=None,
    max_tokens=None,
)
```

在 `agent.run()` 中传入：

```python
result = agent.run(
    task="...",
    max_steps=10,
    history_policy=policy,
    return_state=True,
)
```

`HistoryPolicy.build_query()` 会生成发给 `History` adapter 的检索查询。默认的内部 history 实现会用一个滑动窗口来满足它。

<Tip>
当模型 context 很小，可以收紧 `max_messages`；如果只想保留最近几步，可以设置 `step_window=4`。
</Tip>

## 内置 history 策略

### WindowHistory

`WindowHistory` 是最简单的实现，只保留固定长度的最近消息窗口。

适合：

- run 很短
- 你希望上下文行为完全可预测
- 你不需要自动摘要

### TokenBudgetSummaryHistory

`TokenBudgetSummaryHistory` 会监控 token 预算，并在窗口超出预算时，把更早的消息压缩成一个 summary message。

适合：

- run 已经开始出现明显的 context pressure
- 你只需要一个轻量 summary fallback
- 暂时不需要 `CompactHistory` 的多阶段控制

### CompactHistory

`CompactHistory` 是最完整的内置上下文压缩器，也是 QitOS 长时运行设计最关键的部件之一。

它支持：

- 溢出前 warning thresholds
- 基于 round 的消息分组
- 对巨大旧消息的 micro-compaction
- 对更早历史的 summary compaction
- 可进入 trace 的 compaction metadata

适合：

- 你正在构建真正的 long-running agent
- 你希望 compaction 行为是可检查、可研究的
- 你不想把 context management 藏成黑盒中间件

## 实践中的 context compaction

`CompactHistory` 值得重点理解，因为它体现了 QitOS 的一个重要主张：长时运行行为必须显式且可调试。

它的关键组成包括：

- `CompactConfig`：阈值与 compaction 策略
- `MessageGrouper`：消息如何按轮分组
- micro-compaction：裁剪超长旧消息，但保留预览
- summary compaction：把更早轮次替换成连续摘要

典型配置：

```python
from qitos.kit.history import CompactConfig, CompactHistory

history = CompactHistory(
    config=CompactConfig(
        max_tokens=16000,
        keep_last_rounds=2,
        keep_last_messages=8,
        warning_ratio=0.8,
        auto_compact=True,
    ),
    llm=llm,
)
```

如果你想把 context behavior 也纳入研究 artifact，这通常是最合适的选择。

## Memory

`Memory` 存储的是 `MemoryRecord`（`role`、`content`、`step_id`、`metadata`）。Engine 会在每步后调用 `memory.append()`，并在需要时检索记录，把结果重新注入 `prepare()` 或其他策略层。

四个常用 adapter 都实现同一个 `Memory` 抽象：

```python
from qitos.core.memory import Memory, MemoryRecord

class Memory(ABC):
    def append(self, record: MemoryRecord) -> None: ...
    def retrieve(self, query=None, state=None, observation=None) -> list[MemoryRecord]: ...
    def summarize(self, max_items: int = 5) -> str: ...
    def evict(self) -> int: ...
    def reset(self, run_id=None) -> None: ...
```

### 给 agent 挂载 memory

可以直接把 memory 实例传给 `AgentModule` 构造函数：

```python
from qitos.kit.memory import WindowMemory

agent = MyAgent(
    llm=llm,
    memory=WindowMemory(window_size=20),
)
```

---

## WindowMemory

保留最近 `window_size` 条记录，超出的旧记录会被丢弃。

```python
from qitos.kit.memory import WindowMemory

memory = WindowMemory(window_size=20)
```

适合短期滚动记忆，以及你只关心最近若干 observations 的场景。

## SummaryMemory

保留最近 `keep_last` 条原始记录；当调用 `evict()` 时，更早的记录会被压成 summary 字符串。

```python
from qitos.kit.memory import SummaryMemory

memory = SummaryMemory(keep_last=10)
```

适合：想保留最近原始信息，同时给更早内容保留一份紧凑摘要。

## VectorMemory

使用语义相似度进行检索。每次 `append()` 都会对记录做 embedding，`retrieve()` 时按相似度排序。

```python
from qitos.kit.memory import VectorMemory

memory = VectorMemory(top_k=5)
```

适合：你希望 agent 在长时运行中按语义召回先前发现，而不是只按时间窗口取最近消息。

## MarkdownFileMemory

把所有记录持久化到本地 Markdown 文件，是一种可检查、可跨 session 延续的记忆形式。

```python
from qitos.kit.memory import MarkdownFileMemory

memory = MarkdownFileMemory(
    path="./workspace/memory.md",
    max_in_memory=200,
)
```

适合：长生命周期 agent、多 session 工作流，或者你需要让 memory 变成人类可审阅 artifact 的情况。

## 什么时候该选什么

一个实用的选择顺序通常是：

1. 先只用 state。
2. 当消息窗口不够时，加 `HistoryPolicy`。
3. 当简单裁剪会伤害行为时，加 `CompactHistory`。
4. 当你需要跨步语义召回或持久记忆时，再加 `Memory`。

这四层不是彼此替代，而是不同问题的答案。
