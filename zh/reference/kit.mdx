`qitos.kit` 提供可复用的 agent 编写积木。所有组件都挂接在 `AgentModule + Engine` 主流水线上，不会引入第二个 orchestrator。

```python
from qitos.kit import ReActTextParser, WindowMemory, CodingToolSet, ...
```

如果你正在做 v0.4 的多模型研究，推荐优先查看 `qitos.harness`：

```python
from qitos.harness import build_harness_policy, build_model_for_preset
```

`OpenAICompatibleModel(...)` 仍然适合手工接单一路径；但如果你想让同一个 agent 在多个模型家族之间切换，family preset 才是推荐入口。

## 如何阅读这页

把这页看成 QitOS authoring 的能力地图：

| 如果你在决定…… | 优先查看 |
|---|---|
| 模型该如何输出动作 | Parsers |
| 长时运行如何保住有效上下文 | Memory 与 History / Compaction |
| 要暴露哪些工具 | Tool sets |
| 从哪些现成 registry 开始 | Preset builders |
| 如何加 planning 或 search 结构 | Planning |

## 能力总览

### Parsers

- `ReActTextParser`：解析 `Thought:` / `Action:` 文本协议
- `JsonDecisionParser`：解析 JSON 决策对象
- `XmlDecisionParser`：解析 XML 协议
- `MiniMaxToolCallParser`：处理模型原生 tool-call 结构
- `TerminusJsonParser` / `TerminusXmlParser`：处理带显式终止信号的格式

### Tool presets

- `coding_tools(...)`：标准 coding workspace
- `advanced_coding_tools(...)`：更偏 Claude 风格的 coding preset
- `web_tools()`：web research 与 extraction
- `task_tools(...)`：持久 task-board 工作流
- `security_audit_tools(...)`：防御式仓库审计
- `thinking_tools()`：显式 thought-recording 流程

### 长时上下文能力

- `WindowHistory`
- `TokenBudgetSummaryHistory`
- `CompactHistory`
- `WindowMemory`
- `SummaryMemory`
- `VectorMemory`
- `MarkdownFileMemory`

### Planning 与 Search

- `NumberedPlanBuilder`
- `PlanCursor`
- `DynamicTreeSearch`

<Tab title="Parsers">

parser 负责把原始 LLM 输出转换成 `Decision`。最重要的原则是：**parser 必须与 prompt 请求的输出格式一一对应。**

<AccordionGroup>
<Accordion title="ReActTextParser">

用于解析带 `Thought:`、`Action:`、`Final Answer:` 标记的 ReAct 文本。

```python
from qitos.kit import ReActTextParser
```

当你的 prompt 要求模型输出文本 ReAct 协议时，就使用它。

</Accordion>

<Accordion title="JsonDecisionParser">

用于解析 JSON 输出。适合需要更严格结构化契约的场景。

```python
from qitos.kit import JsonDecisionParser
```

</Accordion>

<Accordion title="XmlDecisionParser">

用于解析 XML 输出。适合需要 XML 风格结构控制的模型或协议。

```python
from qitos.kit import XmlDecisionParser
```

</Accordion>

<Accordion title="Terminus parsers">

`TerminusJsonParser` 与 `TerminusXmlParser` 用于带显式终止标记的协议。适合终端控制、长时交互等更强约束场景。

</Accordion>
</AccordionGroup>

</Tab>

<Tab title="History & Compaction">

QitOS 把长时上下文管理当成一等能力，而不是隐形中间件。

- `WindowHistory`：最简单的最近窗口
- `TokenBudgetSummaryHistory`：超 token budget 时自动摘要
- `CompactHistory`：多阶段 compaction，支持 warning、round grouping、summary compaction 与可观测 metadata

如果你在研究 long-running agents，`CompactHistory` 往往是最值得重点理解的组件。

</Tab>

<Tab title="Memory">

memory adapters 负责把跨步信息存起来，并在需要时重新检索：

<AccordionGroup>
<Accordion title="WindowMemory">

保留最近 N 条记录。

</Accordion>

<Accordion title="SummaryMemory">

保留最近记录，并把更早记录压缩成 summary。

</Accordion>

<Accordion title="VectorMemory">

按语义相似度检索历史记录，适合长时研究与语义召回。

</Accordion>

<Accordion title="MarkdownFileMemory">

把 memory 写入本地 Markdown 文件，适合持久化与人工审阅。

</Accordion>
</AccordionGroup>

</Tab>

<Tab title="Tool Sets">

QitOS 的 toolsets 帮你直接组合出一个工作环境，而不是一组松散工具。

- `CodingToolSet`：文件、编辑、shell、任务、notebook 等 coding 环境
- `TaskToolSet`：todo / progress tracking
- `SecurityAuditToolSet`：代码与配置安全审计
- 其他更窄场景的 bundles：`report_tools(...)`、`notebook_tools(...)`、`epub_tools(...)`

一般建议：

- 学框架时手工 registry
- 做真实 workflow 时切换到 preset toolset

</Tab>

<Tab title="Planning">

planning helpers 帮你在不重写 runtime 的情况下引入显式计划与搜索结构：

- `NumberedPlanBuilder`：从编号列表构建计划
- `PlanCursor`：记录计划执行进度
- `DynamicTreeSearch`：在分支候选之间做选择与扩展

</Tab>
</Tabs>
