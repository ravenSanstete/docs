Every time you run an agent, QitOS writes a self-contained trace directory. Tracing is on by default. It gives you a complete, structured record of what the agent did, step by step, including decisions, tool calls, state diffs, and stop conditions.

## Why traceability is a first-class feature

QitOS is designed for research and long-running agent work, so tracing is not an afterthought.

The trace system exists so you can:

- replay what the agent actually did instead of relying on anecdotes
- compare runs across prompts, models, and tool configurations
- debug parser failures, bad tool choices, and context collapse
- export a run as an artifact for collaborators or audits

This is why QitOS ships not just trace files, but also `qita` as a built-in traceability interface.

## What you get per run

Each run produces a directory at `<trace_logdir>/<run_id>/` containing three files:

| File            | Contents                                                                              |
| --------------- | ------------------------------------------------------------------------------------- |
| `manifest.json` | Run metadata: `run_id`, `status`, `model_id`, `step_count`, `stop_reason`, `final_result`, timestamps, and config hashes. |
| `events.jsonl`  | One JSON line per lifecycle event (`INIT`, `DECIDE`, `ACT`, `REDUCE`, `CHECK_STOP`, `END`, etc.), each with a `phase`, `step_id`, `ok` flag, `payload`, and `ts`. |
| `steps.jsonl`   | One JSON line per completed step, containing the observation, decision, actions, action results, tool invocations, critic outputs, and state diff. |

The `manifest.json` is written at run start with `status: "running"` and updated at the end with `status: "completed"` or `status: "failed"`.

## Run IDs and directory layout

By default, `agent.run()` generates a run ID from your agent's name and a UTC timestamp:

```
{agent.name}_{YYYYMMDD_HHMMSS_ffffff}
```

For example: `research_agent_20260407_143022_819441`.

The trace is written to:

```
./runs/
  research_agent_20260407_143022_819441/
    manifest.json
    events.jsonl
    steps.jsonl
```

You can change the base directory with `trace_logdir` and control the ID prefix with `trace_prefix`:

```python
result = agent.run(
    "summarize the dataset",
    trace_logdir="./experiments",
    trace_prefix="exp_42",
)
# writes to ./experiments/exp_42_20260407_143022_819441/
```

## TraceWriter

`TraceWriter` is the component that performs the actual I/O. You do not need to instantiate it manually — `agent.run()` creates one when `trace=True` (the default).

If you need to supply a pre-configured writer (for example, to set a fixed `run_id` for reproducibility), pass it directly:

```python
from qitos.trace import TraceWriter

writer = TraceWriter(
    output_dir="./runs",
    run_id="fixed_run_abc123",
    metadata={"model_id": "gpt-4o"},
)

result = agent.run("compute something", trace=writer)
```

`TraceWriter` constructor parameters:

| Parameter        | Type               | Description                                                            |
| ---------------- | ------------------ | ---------------------------------------------------------------------- |
| `output_dir`     | `str`              | Base directory. The run directory is created as `output_dir/run_id/`. |
| `run_id`         | `str`              | Unique identifier for this run.                                        |
| `schema_version` | `str`              | Trace schema version. Defaults to `"v1"`.                              |
| `metadata`       | `dict \| None`     | Extra fields stored in the manifest (`model_id`, `seed`, etc.).        |
| `strict_validate`| `bool`             | If `True`, validates artifacts against the schema on finalization. Defaults to `True`. |

## Disabling tracing

Pass `trace=False` to suppress all trace output:

```python
result = agent.run("quick test", trace=False)
```

<Note>
  When `trace=False`, no directory is created and no files are written. `EngineResult.events` and `EngineResult.records` are still populated in memory — they just are not persisted to disk.
</Note>

## Viewing traces with `qita`

QitOS ships a CLI called `qita` for inspecting trace directories:

```bash
# Overview of all runs in the default ./runs directory
qita board --logdir ./runs

# Replay a specific run step by step
qita replay --run ./runs/research_agent_20260407_143022_819441

# Export a run to a standalone HTML file
qita export --run ./runs/research_agent_20260407_143022_819441 --html run.html
```

For a full walkthrough of the observability tooling, see **Guides > Observability**.

## What qita adds on top of raw trace files

The raw files are the source of truth. `qita` is the inspection layer on top of them.

- `qita board` helps you compare many runs at once
- `qita replay` helps you understand one run as a temporal process
- `qita export` helps you share a run without requiring a local server

In practice, this means QitOS gives you both machine-readable traces and a human-friendly review surface.

## Reproducibility requirements

The trace system is designed for research reproducibility. A valid trace must include the following fields in `manifest.json`:

- `task_id` and `task_hash`
- `model_name`
- `config_hash`
- `tool_manifest`

These are populated automatically from the `Task` object and the `TraceWriter` metadata when `strict_validate=True`. If any required field is missing at finalization, a validation error is raised.

<Tip>
  Pass `model_id` in the `TraceWriter` metadata so it appears in `manifest.json`. When using `agent.run()` with tracing enabled, QitOS reads `agent.llm.model` automatically.
</Tip>
